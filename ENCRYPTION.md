# 🔒 Шифрование в Nimbus Messenger

## Обзор

Nimbus использует многоуровневое шифрование для защиты ваших данных:

### 1. Шифрование сообщений (AES-256-GCM)
- **Что шифруется**: Все текстовые сообщения в базе данных
- **Алгоритм**: AES-256-GCM (военный стандарт)
- **Ключ**: 64-символьный hex-ключ (32 байта)
- **Где хранится**: В `.env` файле сервера (`ENCRYPTION_KEY`)

### 2. Шифрование файлов
- **Что шифруется**: Все загруженные файлы (изображения, видео, аудио, документы)
- **Алгоритм**: AES-256-GCM
- **Расшифровка**: Происходит автоматически при скачивании
- **Прозрачность**: Пользователи не замечают шифрования

### 3. HTTPS/TLS (опционально)
- **Что шифрует**: Весь трафик между клиентом и сервером
- **Протокол**: TLS 1.3
- **Статус**: Опционально для локальной разработки, обязательно для production

## Настройка

### Базовая настройка (уже включена)

Файлы уже зашифрованы благодаря `ENCRYPTION_KEY` в `.env`:

```env
ENCRYPTION_KEY=36427af4278b7198dc850c7235c4c85feda7275d89fe3d360c79a1af94579765
```

### Включение HTTPS (для production)

1. **Сгенерируйте SSL сертификат**:
```bash
# Для production используйте настоящий сертификат (Let's Encrypt, Comodo, etc.)
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
  -keyout cert.key -out cert.crt \
  -subj "/C=US/ST=State/L=City/O=Organization/CN=localhost"
```

2. **Обновите `.env`**:
```env
USE_HTTPS=true
HTTPS_KEY_PATH=./cert.key
HTTPS_CERT_PATH=./cert.crt
HTTPS_PORT=3443
```

3. **Перезапустите сервер**

## Важные предупреждения

### ⚠️ Потеря ключа шифрования
Если вы потеряете `ENCRYPTION_KEY`:
- **Все зашифрованные сообщения станут нечитаемыми**
- **Восстановление невозможно**
- **Сделайте резервную копию ключа!**

### 🔄 Смена ключа шифрования
Для смены ключа:
1. Создайте резервную копию базы данных
2. Обновите `ENCRYPTION_KEY` в `.env`
3. Перезапустите сервер
4. **Старые сообщения не расшифруются новым ключом!**

## Проверка статуса

### Проверить включено ли шифрование
```bash
curl http://localhost:3001/api/health
```

В логах сервера должно быть:
```
🔒 Шифрование сообщений включено (AES-256-GCM)
```

### Проверить HTTPS
```bash
curl -k https://localhost:3443/api/health
```

## Безопасность

### Рекомендации для production:
1. ✅ Используйте HTTPS обязательно
2. ✅ Храните `ENCRYPTION_KEY` в secure vault (не в .env)
3. ✅ Регулярно обновляйте SSL сертификаты
4. ✅ Используйте HSTS заголовки
5. ✅ Включите Perfect Forward Secrecy
6. ✅ Ограничьте доступ к базе данных
7. ✅ Регулярно делайте backup ключей

### Генерация безопасного ключа:
```bash
# Сгенерировать новый 64-символьный hex ключ
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"
```

## Поддержка

При проблемах с шифрованием:
1. Проверьте логи сервера
2. Убедитесь что `ENCRYPTION_KEY` задан
3. Проверьте права доступа к файлам
4. Убедитесь что сервер имеет доступ к базе данных
